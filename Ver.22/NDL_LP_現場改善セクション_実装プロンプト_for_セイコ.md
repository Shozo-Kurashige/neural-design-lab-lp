## 参考資料について（重要）

このタスクの検討過程で「見積システム進化ロードマップ」「建設会社向けLP」の2つのHTML/TSXファイルを参照しましたが、**今回は添付しません**。理由は、両ファイルとも配色（赤・青・黄・緑の多色、RAG/AI訴求の強いコピー）が今回の方針と逆であり、コードを見せると構造ごと引っ張られるリスクがあるためです。

今回参考にしてよい考え方は次の3点のみです。コード・配色・コンポーネント構造は一切踏襲しないでください。

- As-Is / To-Be による変化の可視化
- 整理 → 共有 → 継承 という段階的な進行
- 現場の知恵を会社の資産に変えるというストーリー

配色・部品・余白・アニメーションは、すべて**このプロジェクトの既存Hero/Challenges/Approachセクション**から取得してください。外部ファイルではなく、既存コードが唯一の正とします。

## 目的

既存LPの `philosophy` セクション直後に、NDLが現場をどう変えるかを具体的に伝えるセクションを追加します。

モデルケース：**社長しか扱えない見積Excelを、現場と会社の資産へ変える事例**

このセクションの目的は見積システムの販売ではありません。NDLが「今の業務を否定しない」「いきなりAI化しない」「現場の知恵を整理する」「チームで使える状態にする」「判断や例外対応を資産として残す」という順序で改善を進める会社であることを伝えます。

## セクションの位置づけ（表現ルール）

「導入事例」「実績紹介」と断定しないこと。実在案件の成果を保証する表現も避けること。

- 最優先表記：**現場改善の一例**
- 代替候補：変化のモデルケース／NDLが設計する業務改善の一例

## セクション構成（2層）

### 1. 変化のフロー（Before → 3ステップ → After）

**修正点（実装後のフィードバック反映）**：前回の実装ではBefore/Afterが横並び、3ステップがその下という3ブロック構成になっており、「変化の流れ」が一目で伝わらなかったため、以下の通り再構成してください。

- 3ステップは、Before/Afterの下ではなく、**両者の間**に配置する
- Before → 3ステップ → After を、矢印または連続するラインで一本につないだ流れとして視覚的に接続する（Before各項目と各ステップを1対1で対応させる必要はなし。全体として一続きの流れに見えれば十分）
- Afterのカードは、Beforeよりわずかに明るい配色で強調する（詳細は後述のデザイン制約を参照）
- Before/Afterの項目数は、それぞれ4→3項目に削減する

**PC構成（横一列・3ブロック）**：
```
[ BEFORE カード ] → [ 01 → 02 → 03（縦積み）] → [ AFTER カード ]
```
中央に3ステップを縦に並べて配置し、Before/Afterの2カードで挟む形にする。Before→ステップ、ステップ→Afterの間を矢印またはラインで接続する。

**スマホ構成（縦一列）**：
```
BEFORE
  ↓
01 → 02 → 03
  ↓
AFTER
```
上から下へ、縦の連続ラインで接続する。

**BEFORE（現状・3項目）**
- 複雑な関数が絡み合い、全体を把握できる人が限られている
- 見積の作成や修正が、社長や特定の担当者に集中している
- 難工事や例外対応の判断理由が記録されず、毎回ゼロから判断している

**01 整理する**
業務の流れ、入力項目、計算ロジック、単価情報を見える化します。

**02 共有する**
特定の人だけが扱っていた仕組みを、チームで使える状態に整えます。

**03 継承する**
判断理由や例外対応を蓄積し、将来の検索やAI活用につなげます。

**AFTER（改善後・3項目）**
- 入力、計算、単価情報の役割が整理され、仕組みを理解しやすくなる
- チームで見積を作成・確認し、必要な判断だけを社長が行える
- 例外対応の理由が蓄積され、類似案件の判断に再利用できる

※「単価・材料情報の一元化」の項目は、他2項目と趣旨が重なるため削減対象としました。

Before/Afterの文章量は維持してください。極端に短いキャッチコピーへ置き換えず、業務の変化が具体的に分かる文量を保つこと。派手なロードマップ表現・多色展開は禁止（矢印・連続ライン・番号進行の表現は使用可）。

### 2. まとめメッセージ

セクション下部。結論として適度に強調（CTAボタンは今回追加しない。CTA設計は後工程）。

```
ツールを入れることが、ゴールではありません。

現場にある知恵を整理し、誰か一人に頼らなくても、
業務と判断が無理なく続いていく状態をつくります。
```

## デザイン制約

- 既存Hero/Challenges/Approachの角丸・枠線・影・余白・見出しサイズ・本文サイズ・セクション幅・背景色・ホバー表現をそのまま踏襲。新しいデザインシステムを作らない。
- 色は既存のCSS変数／Tailwind設定のみ使用。新色を追加しない。
- Beforeは既存カラーの低彩度・グレー・薄い背景。Afterはブランドカラーで前向きな変化を表現。
- Afterのカードは、Beforeとの対比がひと目でわかる程度に、既存ブランドカラーの明度・彩度をやや高めて強調する（背景の着色を少し濃くする、またはアクセントの縁取りを加える程度）。新色の追加や派手な発光表現は不可。
- 赤・青・黄・緑をフェーズごとに使い分ける構成は禁止（多色化しない）。
- アイコンは既存ライブラリがあればそれを使用。なければ既存と整合するシンプルなSVG。絵文字禁止。

## アニメーション制約

既存Hero/Challenges/Approachと同じ速度・イージング・発火条件を使用。新規セクションだけ別方式にしない。

- 見出し：スクロール進入時に軽いフェードアップ
- Before/After：PCはわずかな時間差で表示、スマホは上から順に自然表示
- 3ステップ：01→02→03の順に軽い遅延

禁止：強いバウンド、大きな拡大縮小、高速な横移動、長時間ループ、常時点滅、読了前の要素移動、スクロール操作の妨げ。`prefers-reduced-motion` 有効時はアニメーションを停止・簡略化。

## レスポンシブ

- **PC**：Beforeカード｜3ステップ（縦積み）｜Afterカードの3ブロックを横一列に配置し、矢印またはラインで接続。Before/Afterのカード高さを可能な範囲で揃える、最大幅は既存セクションと統一
- **タブレット**：横幅に応じて3ブロックの横並びを維持するか、収まらない場合は縦方向のフロー（Before→01→02→03→After）へ切り替え。不自然な折り返しを避ける
- **スマホ**：Before → 01 → 02 → 03 → After の順で縦一列のフローとして配置（縦積み固定）。カード内の左右2分割はしない、横スクロール発生させない、最低16px相当の可読性

## コンポーネント設計（例・過剰分割はしない）

```
TransformationExampleSection
BeforeAfterCard
TransformationStep
```

データ部分は配列化してよい。既存の命名規則に合わせること。

## アクセシビリティ

- セマンティックな `section` / `h2` / `h3`、見出し階層を崩さない
- 装飾アイコンに `aria-hidden`
- 意味を色だけで区別しない、本文と背景のコントラスト確保
- キーボード操作が必要なUIを新設しない
- アニメーションなしでも情報が欠落しない構造

## 完成条件

**内容**：見積Excelの改善例と分かる／見積システム専門会社に見えない／NDLの改善方法のモデルケースとして読める／BeforeがNDLの3ステップ（整理→共有→継承）を通じてAfterへ変わる流れが一目で伝わる／見出し直下の一文コピーで変化の要点が端的に伝わる／AI導入を急がず業務整理を優先する思想が伝わる／実績・成果を過大に断定していない

**デザイン**：既存LPと違和感なくつながる／ロードマップ資料を貼り付けた印象がない／多色で派手にならない／PC・スマホで読みやすい／単体で目立ちすぎない

**動作**：横スクロールなし／既存アニメーションと干渉しない／既存セクションの表示・挙動を壊さない／`prefers-reduced-motion` 対応／コンソールエラーなし／レイアウトシフトが目立たない
